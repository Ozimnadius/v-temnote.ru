# Этапы Верстки И Натяжки

## Назначение

Этот документ описывает разделение работ между статической версткой на Vite и последующей натяжкой готовой верстки на Bitrix.

Цель разделения — сначала быстро и чисто собрать интерфейс по Figma, а затем отдельным этапом интегрировать готовую HTML/SCSS/JS-структуру в Bitrix-шаблон и динамические данные.

## Обязательные Правила

Перед началом работы агент должен учитывать:

- `AGENTS.md`;
- `docs/layout-developer/README.md`;
- `docs/layout-developer/html-structure.md`;
- `docs/layout-developer/scss-structure.md`;
- `docs/layout-developer/figma-to-project.md`.

Если блок относится к типовой структуре, например слайдеру, дополнительно читать правила из `docs/layout-developer/block-types`.

Физические изменения файлов выполняются только после согласования по `AGENTS.md`.

## Этап 1: Vite-Верстка

Используется, когда нужно сверстать страницу или блок по Figma без привязки к Bitrix-компонентам и backend-данным.

На этом этапе агент делает:

- перенос Figma в HTML/SCSS/JS;
- статическую структуру страницы или блока;
- реальные тексты, кнопки, карточки и изображения из Figma;
- адаптивную HTML-структуру;
- слайдеры и интерактив на frontend-уровне;
- подготовку изображений и assets для будущей интеграции.

На этом этапе агент не делает:

- подключение Bitrix-компонентов;
- работу с инфоблоками;
- backend-логику;
- PHP-шаблоны компонентов;
- замену статичных данных на динамические;
- изменение ядра Bitrix.

Даже если верстка выполняется в отдельной Vite-среде, она должна соблюдать проектные правила:

- BEM-именование;
- `.container`;
- порядок внутренних блоков страницы `page__zone > .container > block`;
- один BEM-блок — один SCSS-файл;
- подключение SCSS-блоков через индекс;
- правила изображений `picture` / `img`;
- правила типовых блоков из `docs/layout-developer/block-types`.

Физический путь к SCSS зависит от этапа. В отдельной Vite-среде можно использовать ее директорию исходников, но структура должна оставаться эквивалентной проектной: отдельная папка блоков, один файл на BEM-блок и общий индекс подключений. На этапе Bitrix-натяжки целевой путь для блоков — `local/templates/template/css/blocks`.

Vite не должен становиться источником несовместимой архитектуры. Верстка должна быть готова к переносу в Bitrix без переписывания BEM-структуры.

## Этап 2: Подготовка К Натяжке

Используется после завершения статической верстки и перед переносом в Bitrix.

На этом этапе агент должен проверить:

- какие HTML-фрагменты остаются статичными;
- какие элементы станут Bitrix-компонентами;
- какие списки, карточки, меню, баннеры и изображения станут динамическими;
- какие assets нужно перенести в шаблон;
- какие JS-инициализации должны остаться общими;
- какие классы используются для JS и не должны быть переименованы;
- какие SCSS-файлы нужно подключить в Bitrix-шаблоне.

Результат этапа — понятная карта переноса, а не новая система планов в MD-файлах. Создавать `PLANNING.md`, `docs/plans` или отдельные планы страниц/блоков нельзя.

## Этап 3: Bitrix-Натяжка

Используется, когда готовую верстку нужно интегрировать в текущий Bitrix-шаблон.

На этом этапе агент делает:

- перенос HTML в нужный файл проекта или шаблон компонента;
- подключение собранных CSS/JS assets;
- замену статичных повторяющихся элементов на данные Bitrix, если это входит в задачу;
- сохранение BEM-классов и структуры из верстки;
- проверку совместимости с `header`, `footer`, `wrapper` и текущими шаблонными зонами.

На этом этапе нельзя без отдельного согласования:

- менять базовый `wrapper`;
- менять header или footer;
- переименовывать BEM-блоки, созданные на этапе верстки;
- смешивать натяжку с новой pixel-perfect доработкой, если это не входит в задачу;
- переписывать frontend-структуру без причины.

## Vite И Bitrix

Vite используется как frontend-сборщик и dev-среда для верстки. Он не заменяет Bitrix и не должен брать на себя backend-роль.

Правильное разделение:

- Bitrix/PHP отвечает за HTML, шаблоны, компоненты и данные на этапе натяжки.
- Vite отвечает за сборку frontend assets: SCSS, JS, изображения и dev-HMR.
- В production Bitrix подключает собранные assets, а не Vite dev server.

Если Vite подключается к Bitrix во время разработки, агент должен использовать backend integration-подход: backend отдает страницу, Vite обслуживает frontend assets.

## Выбор Этапа

Если пользователь просит “сверстать по Figma”, агент должен уточнить этап:

1. Vite-верстка.
2. Подготовка к натяжке.
3. Bitrix-натяжка.

Если пользователь явно говорит про статическую верстку, Vite, pixel-perfect или перенос Figma без backend-данных, выбирать этап 1.

Если пользователь говорит про интеграцию в Bitrix, компоненты, инфоблоки, шаблоны компонентов или динамические данные, выбирать этап 3.

Если статическая верстка уже готова, но непонятно, как переносить ее в Bitrix, выбирать этап 2.

## Чеклист

Перед завершением работы агент должен проверить:

- выбран правильный этап;
- правила HTML и SCSS проекта соблюдены;
- перенос из Figma не создает отдельные MD-планы;
- Vite-верстка не содержит решений, которые мешают Bitrix-натяжке;
- Bitrix-натяжка не ломает готовую BEM-структуру;
- изображения подготовлены в выбранном режиме `picture` или `img`;
- типовые блоки, например слайдеры, соответствуют своим спекам.
