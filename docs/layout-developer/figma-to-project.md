# Перенос Figma В Проект

## Назначение

Этот документ описывает правила переноса Figma-дизайна в Bitrix-шаблон проекта.

Спецификация заменяет отдельные MD-планы для страниц и внутренних блоков. Для переноса из Figma агент выбирает одну из двух веток работы:

- общий скелет страницы без внутренних блоков;
- страница с внутренними BEM-блоками и контентом из Figma.

Отдельные MD-планы для страниц, секций и внутренних блоков не создаются.

## Обязательные Предварительные Правила

Перед переносом Figma в проект агент должен прочитать и учитывать:

- `AGENTS.md`;
- `docs/agents/README.md`;
- `docs/agents/html-structure.md`;
- `docs/agents/scss-structure.md`.

Любые физические изменения файлов выполняются только после согласования по правилам `AGENTS.md`.

Для работы с Figma нужно получать данные из Figma MCP:

- metadata или page structure для определения крупных зон страницы;
- Figma context конкретного node для генерации внутренних блоков и переноса контента.

## Выбор Ветки Работы

### Ветка 1: Общий Скелет Страницы

Используется, когда нужно сначала разложить страницу на крупные зоны без детализации внутренних блоков.

Выбирать эту ветку, если задача звучит как:

- создать верхний уровень страницы;
- подготовить структуру секций;
- разложить страницу по крупным зонам;
- сделать SCSS-скелет страницы без контента секций.

### Ветка 2: Страница С Внутренними Блоками

Используется, когда нужно сразу наполнить крупные зоны самостоятельными BEM-блоками с контентом из Figma.

Выбирать эту ветку, если задача звучит как:

- сверстать страницу по Figma со структурой секций;
- создать конкретный блок по Figma node;
- перенести тексты, карточки, кнопки, изображения и порядок элементов;
- наполнить крупные зоны внутренними BEM-блоками.

Если пользователь явно не указал ветку, агент должен уточнить режим работы или предложить начать с ветки 1 для крупной страницы.

## Ветка 1: Общий Скелет Страницы

Порядок работы:

1. Получить metadata или page structure из Figma.
2. Определить верхний BEM-блок страницы, например `main`.
3. Определить крупные зоны страницы по Figma.
4. Создать только элементы верхнего блока страницы:
   - `main__hero`;
   - `main__products`;
   - `main__categories`;
   - `main__offers`;
   - и другие крупные зоны.
5. Создать SCSS-файл страницы, например `_main.scss`, только со скелетом элементов.
6. Подключить SCSS-файл через `bitrix/templates/template/css/blocks/_index.scss`.

В этой ветке нельзя:

- создавать внутренние блоки `main-hero`, `main-products`, `main-categories` и подобные;
- переносить детальный контент секций;
- создавать карточки, кнопки, списки и вложенную структуру секций;
- описывать финальные CSS-свойства.

Пример HTML:

```html
<main class="main">
  <div class="main__hero"></div>
  <div class="main__products"></div>
  <div class="main__categories"></div>
</main>
```

Пример SCSS:

```scss
@use "../helpers/index" as *;

/* main */

.main {
  &__hero {
  }

  &__products {
  }

  &__categories {
  }
}
```

## Ветка 2: Страница С Внутренними Блоками

Порядок работы:

1. Начать с верхнего скелета страницы из ветки 1.
2. Для каждой крупной зоны получить Figma context по ее node.
3. Внутри зоны страницы создать самостоятельный BEM-блок:
   - внутри `main__hero` создать `main-hero`;
   - внутри `main__products` создать `main-products`;
   - внутри `main__offers` создать `main-offers`;
   - и так далее.
4. Перенести в HTML реальный контент из Figma.
5. Для каждого внутреннего BEM-блока создать отдельный SCSS-файл со скелетом.
6. Подключить новые SCSS-файлы через `bitrix/templates/template/css/blocks/_index.scss`.

Внутренний блок не должен быть пустой заглушкой. Его HTML должен отражать структуру и контент соответствующего Figma node.

Для внутренних блоков страницы `.container` ставится сразу внутри зоны страницы: `page__zone > .container > block`. Внутри BEM-блока уже можно создавать элемент `block__container` для внутренней раскладки.

Если дизайн показывает ограниченную ширину контента, `.container` обязателен. Full-width структура допустима только если Figma явно показывает блок на всю ширину.

Отдельные MD-планы для `main-hero`, `main-products`, `main-offers` и других внутренних блоков не создаются.

### Сценарий: Внутренний Блок В Готовом Скелете

Если верхний скелет страницы уже создан по ветке 1, агент может реализовать отдельный внутренний блок внутри готовой зоны.

Порядок работы:

1. Открыть текущий HTML страницы.
2. Найти нужную родительскую зону, например `main__hero`.
3. Проверить, что зона соответствует Figma node нужной секции.
4. Получить Figma context конкретного node.
5. Если дизайн ограничен по ширине, добавить или использовать `.container` внутри зоны.
6. Создать внутри `.container` самостоятельный BEM-блок, например `main-hero`.
7. Перенести в HTML реальный контент из Figma.
8. Создать или дополнить SCSS-файл внутреннего блока.
9. Подключить SCSS-файл через `bitrix/templates/template/css/blocks/_index.scss`, если он еще не подключен.

В этом сценарии нельзя:

- пересоздавать верхний блок страницы, например `main`;
- менять порядок и названия уже созданных `main__...` зон без отдельного согласования;
- наполнять соседние зоны, если задача относится только к одному блоку;
- оставлять внутренний блок пустым;
- создавать отдельный MD-план для внутреннего блока.

Пример HTML:

```html
<main class="main">
  <div class="main__hero">
    <div class="container">
      <section class="main-hero">
        <div class="main-hero__container">
          <h1 class="main-hero__title">
            Велосипеды для движения в городе
          </h1>
        </div>
      </section>
    </div>
  </div>
</main>
```

Пример SCSS:

```scss
@use "../helpers/index" as *;

/* main-hero */

.main-hero {
  &__container {
  }

  &__title {
  }
}
```

## Перенос Контента Из Figma

Во второй ветке обязательно переносить:

- реальные заголовки и тексты;
- подписи кнопок и ссылок;
- карточки и повторяющиеся элементы;
- табы, бейджи и навигацию;
- изображения, иконки и SVG как Figma asset URLs или placeholders с привязкой к node;
- порядок элементов как в дизайне.

Если Figma node содержит повторяющиеся карточки, в HTML нужно отразить структуру повтора и реальный контент видимых карточек.

Если изображение или иконка пока не выгружены в проект, использовать понятный placeholder с привязкой к Figma node или asset URL. Не придумывать несуществующие изображения.

### Перенос Изображений

Перед генерацией HTML с изображениями нужно определить режим вставки:

1. `picture` — адаптивные изображения под разные разрешения.
2. `img` — одно изображение без адаптивных вариантов.

Если задача запускается через skill `layout-developer` / `Верстальщик`, агент обязан спросить пользователя, как вставлять изображения. Без ответа пользователя нельзя молча выбирать режим.

Если выбран `picture`:

- запросить у пользователя Figma-ссылки на версии блока или изображения для других разрешений, если они есть;
- получить Figma context или assets по каждому предоставленному node;
- использовать структуру `<picture>` с `source` для альтернативных разрешений и fallback `img`;
- если дополнительные Figma nodes не предоставлены, не придумывать адаптивные изображения.

Если выбран `img`:

- использовать одно изображение из основного Figma node;
- не создавать `source` и дополнительные адаптивные варианты;
- не заменять выбранный `img` на `picture` без отдельного согласования.

Примеры и базовые HTML-правила для изображений описаны в `docs/agents/html-structure.md`.

Если задача только про структуру, не переносить:

- финальные CSS-свойства;
- pixel-perfect размеры;
- JS-поведение;
- интеграцию с Bitrix-компонентами;
- реальные динамические данные, если они не заданы отдельной задачей.

## Типы Блоков

Если Figma-блок относится к повторяющемуся типу, агент должен учитывать отдельную спецификацию типа блока.

Доступные типы описаны в `docs/agents/block-types/README.md`.

Для слайдеров, каруселей, блоков со slides, pagination, точками, стрелками или carousel-поведением использовать `docs/agents/block-types/slider.md`.

## Правила SCSS-Скелетов

SCSS создается по правилам `docs/agents/scss-structure.md`.

Базовые требования:

- один самостоятельный BEM-блок равен одному SCSS-файлу;
- файлы блоков лежат в `bitrix/templates/template/css/blocks`;
- имя файла соответствует BEM-блоку;
- первая строка файла всегда `@use "../helpers/index" as *;`;
- после `@use` идет комментарий с названием BEM-блока;
- элементы пишутся через `&__element` внутри родителя;
- файл подключается в `bitrix/templates/template/css/blocks/_index.scss` через `@forward`.

Пример:

```scss
@use "../helpers/index" as *;

/* main-offers */

.main-offers {
  &__container {
  }

  &__list {
  }

  &__item {
  }
}
```

Если задача только про структуру, SCSS должен оставаться скелетным. Финальные свойства, адаптив и hover-состояния добавляются отдельной задачей.

## Запреты

Запрещается:

- создавать `PLANNING.md`;
- создавать папку `docs/plans`;
- создавать отдельные MD-планы для страниц;
- создавать отдельные MD-планы для внутренних блоков;
- переносить React, Tailwind или технический код из Figma context напрямую без адаптации к проекту;
- использовать внутренние блоки без родительской зоны страницы;
- создавать пустые внутренние блоки во второй ветке;
- придумывать контент, которого нет в Figma;
- заменять реальные тексты из Figma на `...`, если контент доступен;
- использовать inline-стили в HTML;
- использовать прямые `@media` вместо helpers проекта;
- использовать прямой `:hover` вместо mixins проекта;
- менять header, footer или базовый `wrapper` без отдельного согласования.

## Чеклист Проверки

Перед завершением работы агент должен проверить:

- выбрана правильная ветка переноса;
- `docs/agents/html-structure.md` учтен;
- `docs/agents/scss-structure.md` учтен;
- отдельные MD-планы для страниц и внутренних блоков не созданы;
- HTML-теги выбраны по семантике;
- верхний BEM-блок страницы создан корректно;
- крупные зоны страницы оформлены как элементы верхнего блока;
- внутренние блоки страницы используют порядок `page__zone > .container > block`, если дизайн ограничен по ширине;
- во второй ветке внутренние блоки являются самостоятельными BEM-блоками;
- во второй ветке внутренние блоки содержат реальный контент из Figma;
- если в переносимом Figma node есть изображения, выбран режим вставки `picture` или `img`;
- если выбран `picture`, использованы предоставленные Figma nodes/assets для других разрешений или явно оставлен только доступный источник;
- если внутренний блок добавляется в готовый скелет, соседние зоны страницы не изменены без согласования;
- если блок является слайдером, используется Swiper и официальная структура `.swiper > .swiper-wrapper > .swiper-slide`;
- каждый новый BEM-блок имеет отдельный SCSS-файл;
- каждый новый SCSS-файл начинается с `@use "../helpers/index" as *;`;
- каждый новый SCSS-файл подключен через `@forward`;
- временная `.demo`-разметка не используется как целевая структура;
- inline-стили не добавлены;
- прямые `@media` и `:hover` не добавлены.
